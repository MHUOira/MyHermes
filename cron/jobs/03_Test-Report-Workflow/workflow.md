## 1. 対象ファイル

`filesets.md`の設定を読み込む。

## 2. 新規コンテンツの定義

- 重複確認ログに記載されていない URL の記事
- または、タイトルが完全に一致しない記事
- 重複確認ログファイルが存在していない場合には、取得できたコンテンツを全て新規コンテンツと見做す。

## 3. レポートの内容と書式（厳守）

`report_definition.md`の設定を読み込む。
調査対象・サイト・レポート書式は `report_definition.md` に従い、`report_definition.md` で指定された内容・書式でレポートを生成する。

## 4. 処理フロー

**品質要件のデフォルトと優先順位**
- `report_definition.md` に以下の値が定義されている場合は、それを優先して使用する：
  - `deep_dive_required`
  - `minimum_excerpt_length`
- 値が未定義または未記載の場合は、以下のデフォルト値を使用する：
  - `deep_dive_required`: false
  - `minimum_excerpt_length`: 500文字

### 1. 重複確認ログ の存在を確認する
   - ローカルに存在する場合はそのまま使用
   - 存在しない場合は空の 重複確認ログ をローカルに作成

### 2. 重複確認ログ の内容を確認して、既に存在するレポートの URL を一覧化

### 3. Web検索で新規事例を抽出（重複確認ログ に記載されていないもの）
#### コンテンツ取得の優先順位:
(1) browser_navigate + browser_snapshot（full=true） を第一選択とする
(2) web_extract は補助的に使用
(3) どうしても browser が不安定な場合は web_extract にフォールバック  
#### 具体的な取得手順:
(1) web_search を使用して、`report_definition.md`で規定した調査対象のサイトまたは調査条件に合致した新規サイトに対して、`report_definition.md`で規定した調査目的のキーワードを検索し、ヒットしたURL を取得
(2) 各 URL に対して browser_navigate を実行
(3) browser_snapshot（full=true）でページ内容を取得
(4) 品質要件に基づく深堀り要否の判断
   - `deep_dive_required: true` の場合：
     - 取得した内容が `minimum_excerpt_length` を満たしていない、または動的展開要素が検出された場合は、(5)の深堀り手順を実行する
   - `deep_dive_required: false` の場合：
     - 初回 snapshot の内容で (8) のステップへ進む（深堀りは行わない）
(5) 十分な長文が取れなかった場合、または snapshot に "use browser_snapshot for full content" / truncated 注記 / "Continue Reading" / "Read more" / "Show more" / "Load more" などの動的ロード要素が検出された場合：
   - browser_snapshot で該当ボタン/リンクの ref ID（@eXX形式）を特定
   - browser_click で「Continue Reading」「Read more」「Show more」「Load more」などのボタンを明示的にクリックして展開
   - 展開後に browser_scroll(down) を3〜5回繰り返し、各回で browser_snapshot(full=true) を再実行
   - 各回後に browser_snapshot(full=true) を再実行して追加コンテンツを取得
   - 上記の展開＋スクロール＋再スナップショットサイクルを**最大5回**まで繰り返す（または十分な長文抜粋が可能になるまで継続）
   - この「クリック＋スクロール＋再snapshot」サイクルを最大5回まで継続（初回snapshotだけで終わらないこと）
   - ページネーション（Next / もっと見る / Load more comments）も browser_click で対応
   - 目標：抜粋原文 `minimum_excerpt_length` 以上を目指す(可能な限り記事を深堀りするが未達でも次のステップへ進める)
   - すべての取得試行（クリックしたref、スクロール回数、取得文字数）を debug.log に記録
(6) 各記事について、抜粋原文の文字数が `minimum_excerpt_length` 以上になることを目指して可能な限り、記事を深堀りするが、未達でも次のステップへ進める。
(7) (5)-(6)のプロセスは必須とし、単なる初回snapshotで終了しない
(8) (7)まで実行した結果が `minimum_excerpt_length` に満たなかった場合には、そこまでの内容を**「部分抽出事例」**として扱う。
#### 失敗時の扱い:
(1) browser_navigate / browser_snapshot が失敗した場合：web_extractへフォールバック
(2) すべての取得試行の履歴（クリックしたref、スクロール回数、取得文字数）を debug.log に記録  
####  `minimum_excerpt_length` 未達の場合の扱い:
そこまでに取得した内容で部分抽出事例レポートを作成する。重複確認ログには記録しない。

### 4. 既存レポートとの重複・競合をチェック
すでに調査済のレポートと重複していた場合には、いつのどのレポートと競合していたかを明示した上でレポートを出力する。
競合情報はレポートの先頭部分、日付の後に入れること。

### 5. レポート内容を生成（指定書式厳守）
   - `report_definition.md`に記載された手順、書式、出力先を厳守
  

### 6. レポートの出力
   - filesets.mdで指定した出力先に出力する
   - レポートを保存
   - 重複確認ログに新しいエントリを追記


## 5. エラーハンドリング（各失敗モードごとの具体的なフォールバック）

**重複確認ログ 関連**
- 重複確認ログ の内容解析に失敗した場合（フォーマット不正など）:
  - 空として扱い、処理を続行（新規作成扱い）
  - ファイルが存在していない場合にはブランクファイルを作成

**ローカルファイル書き込み関連**
- `outputs/YYYY-MM-DD.md` または 重複確認ログ のローカル書き込みに失敗した場合:
  - `error.log` に「[timestamp] LOCAL WRITE FAILED: [詳細]」を追記
  - `error.log`への出力が出来なかった場合にはDeliverへ出力(エラーファイル出力できた場合にはDeliverには出力しない)
  - ユーザーに手動確認を促すメッセージを出力

**新規コンテンツ0件の場合**
- 検索や抽出プロセスで**URLが一切取得できなかった場合**のみ「本日の新規事例: 0件」として扱い、デバッグ情報を出力（[SILENT]禁止）
- 抜粋原文が500文字以上：**「完全事例」**としてレポートを生成し、重複確認ログに記録
- 抜粋原文が500文字未満：取得できた範囲で**「部分抽出事例」**としてレポートを生成するが、重複確認ログには記録しない。
  - 原文抜粋・逐語訳・分析を可能な限り実施（短くても報告）
  - レポートの見出しは「本日の新規事例（部分抽出含む）」とし、どの部分が完全/部分かを明記
- 取れた部分だけでもきちんと報告することを優先する
- 部分抽出の場合も、取得プロセス（クリック・スクロール回数・文字数）の詳細を必須出力項目に追加

**その他予期しないエラー**
- 可能な限り処理を継続
- すべてのエラーは `error.log` にタイムスタンプ付きで記録
- 重大なエラー以外は cron を停止させない

**重複確認ログ のエントリ形式（厳守）**
各行: `YYYY-MM-DD|URL|記事タイトル`（例: 2026-06-19|https://quora.com/...|My cuckoldry experience...）
重複判定は URL と タイトル の完全一致で行う。

## 6. ログ設計

- `重複確認ログ`: 重複管理専用（成功したレポートのみ記録）
- `error.log`: エラー・リトライ履歴専用（別ファイルで管理）
- `debug.log`: debug用に指示した情報の出力(最初に日時)
